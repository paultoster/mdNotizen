Branch naming rules:
 
 
 
  a) Release branch : release/<name>
 
  b) Feature branch : feature/<name>
 
  c) Hotfix branch  : hotfix/<name>
 
  d) Support branch : support/<name>
 
 
 
  - <name> : 5 to 40 alphanumeric characters including dash
 
  - all characters in lowercase
 
 
 
Examples of valid branch names according to the naming rules:
 
 
 
  a) release/brimley-2019
 
  b) feature/trajectory-planner-refactoring
 
  c) hotfix/vehicle-pose-unhandled-exception
 
  d) support/ecal-4-grid
 
 
 
Explanations of the branch naming rules:
 
 
 
  a) Several features are accumulated in this branch. Colloquially it is also called the integration branch in the GRID.
 
  b) Branch in which a feature for a GRID component is developed. In most cases this will be the right branch type for you.
 
  c) Bugfix branch which should flow back into all active release branches and into the master branch.
 
  d) Support branch where legacy components are needed that are no longer used in the current master branch.
 
 
 
The rules are only applied to newly created branches. This means that existing branches that are already on the GitHub server remain unaffected by the new naming convention and can be updated as before. Nevertheless we would appreciate it if the old branch names could be renamed according to the new naming rules. This can be done either with a Git GUI tool or with the Git command line git branch -m <oldname> <newname>.
 
 
 
The new naming rules will be activated on Monday, 11.11.2019 at 11:11 CET. From this point on, a git push of a newly created branch for the GRID Project that does not comply with the above naming convention, will automatically be rejected by the GitHub server.

To make it short:
Please read the handout!

All pushed commits to the remote repository (http://github.conti.de/CTZS/GRID) will trigger an automated build for windows and linux.
If your branch uses the name denomination (*/for/*, explained in the handout) it will also trigger a pull request for the merge into the target (usually master),
after a successful build you can merge your branch into the master.
The existing branches can be kept but you need to merge the master into your branches to get the latest workflow file updates.
The existing branches are mentioned in the handout as "branches with manual creation of pull requests".

You can get an overview of the build status of each branch and/or pull request here: http://jenkinsctzs.conti.de/job/GRID_Lifecycle/job/VerifySourceCode/
IF you cant see anything in the Jenkinsview you need access, please contact your tooling domain.
